`/cptpp` holds the scripts used to *compile* Excel sheets, CSVs, JSON, APIs formats and other raw data types for the calculator. These data represent the tariff schedules, HS codes, countries, and in the future the rule of FTAs itself. This format of the data is auto-detected, then it is cleaned, parsed, validated, then generated to form the Ocaml intermediate representation (IR). These many .ml files (i.e. individual tariff schedules) are then marshalized and serialized to form a binary blob (i.e. complete CPTPP tariff schedules). The binary blob would then be used for `/calc`.

The compiler does not run directly as an independent process. Instead, it would be executed in `/calc` with a `make sync`, where Excel sheets, JSON files, etc. are called as `cptpp file-name -o output-file`. The output files are stored in `/calc/lib/data` where calc would demarshalize the data and use it as if it is a variable in Ocaml.

If you know clang from LLVM, the process and commands are shockingly similar. That's because this module is inspired directly by how clang parse C code to LLVM IR.

## How to run
Run `make setup` followed by `make install` to configure the environments and install the global `cptpp` command.

To get the marshalized version `.bin`, you run it normally: `cptpp one.csv two.json three.xlsx -o output-file.bin`.

To get the Ocaml IR (.ml) for one of the files, run `cptpp --emit-ml file-name -o output-file.ml`. There's nothing special about the Ocaml IR, it is just an autogenerated variable from various python and Ocaml scripts. Currently, the expected file types are JSON, CSV and XLSX.


## Logic
Here, `/cptpp/lib` is tasked with converting a vast array of tariffs and data to an intermediate form (ocaml objects) where /calc (compiling from Ocaml to source code) can understand and manipulate. The most direct comparison is with LLVM's clang converting from C code to LLVM IR (Driver -> Lex -> Parse -> Sema -> AST -> CodeGen). In /cptpp, it is done the same:

1. `driver`: The orchestrator. When you type your commands to compile your .xlsx to Ocaml, you would pass through here. It would handle the different command line arguments and map out the compilation pipeline.
2. `lexer`: Preprocessing. (Basically using Python libraries to parse each of these data in a Python-readable format). Crucially it communicates with `parser` to coordinate the stream of 'tokens' to not overwhelm the machine.
3. `parser`: Syntactic analysis. Here we check for corruption in the .xlsx. The python scripts would flag when the data is corrupted and does not make sense (impossible to generate a proper AST).
4. `ast`: Abstract syntax tree. This is where we use python scripts to write to Ocaml. However, instead of writing the ocaml code directly, we would write the abstract syntax tree in ocaml from the python stream.
5. `codegen`: AST-to-IR Lowering. This is where we actually write the OCaml IR, from the OCaml AST. From this, we use the abstract syntax tree to generate an efficient representation of the IR.
6. `serializer`: Convert to .bin. We use bin_prot to serialize and marshalize OCaml primitives to a binary state while keeping many type-safety features. This is how we ship the tariff schedules/hs code/raw data to `/calc`, where this is the lowest level representation of the data that is language agnostic.

We also enforce a rule here that data cannot flow backward and must flow in the correct order (driver -> lexer -> parser -> ast -> codegen -> serializer). See [this article for how to coordinate between the sections](https://eli.thegreenplace.net/2012/07/05/how-clang-handles-the-type-variable-name-ambiguity-of-cc).

You can call the individual section too: 
1. `driver`: N/A
2. `lexer`: `cptpp --lexer harmonized-system.csv` (give you a list of tokens)
3. `parser`: `cptpp --parser harmonized-system.csv` (give you syntax errors)
4. `ast`: `cptpp --ast harmonized-system.csv` (dump you the abstract syntax tree)
5. `codegen`: `cptpp --emit-ml file-name.csv -o output-file.ml` (one Ocaml file)
6. `serializer`: `cptpp --link vietnam.ml japan.ml -o tariff.bin` (serialized Ocaml .bin)




Note to self:

An even more direct path would be to import the AST directly from python `parser` to ocaml, have a light translator/extractor in ocaml `ast`, then finally do AST calculations in `ast`. Use pyml library.

## Format notes

* For the harmonized system, files use the following format: hs_XXX_YYYY.csv. XXX is 3-letter country codes, YYYY is year of publishing. CSV is convenient and easy to use for OCaml.
* For tariffs, files use the following format: effective_YYYYMMDD.json. YYYYMMDD is the enforcement date. For each change in the tariffs, there is a complete matrix.
* metadata.json is used to add display name (VNM -> "Vietnam") and link files in a systematic way (in metadata.json, "effective_20250103.json" has attribute "date": "2025-01-03")
